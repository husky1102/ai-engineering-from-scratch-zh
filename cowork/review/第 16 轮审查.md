# 第 16 轮审查

日期：2026-06-09

范围：
- 接续第 15 轮后，从自动扫描出的早期合同问题开始检查。
- 重点检查 `00-dev-environment/01-dev-environment` 的中文翻译、quiz 结构、代码入口、demo 退出行为和测试合同。

## 结论

第 16 轮仍有问题，不能结束循环。

`phases/00-setup-and-tooling/01-dev-environment` 是学生入门第一课，但保留旧 quiz 分布、没有 `code/tests/`，并且普通运行验证脚本会在当前环境不满足 Python 3.10+ 或依赖缺失时退出 1。对“配置环境”课程来说，诊断失败是教学输入，不应该让课程 demo 本身失败；需要默认 exit 0，并提供 `--strict` 供 CI 或自检使用。

## Findings

### P1 - 开发环境验证脚本默认退出 1

文件：
- `phases/00-setup-and-tooling/01-dev-environment/code/verify.py`
- `phases/00-setup-and-tooling/01-dev-environment/code/verify.ts`
- `phases/00-setup-and-tooling/01-dev-environment/code/main.rs`

证据：
- `python3 phases/00-setup-and-tooling/01-dev-environment/code/verify.py` 在当前 Python 3.9.6 环境下退出 1。
- `rustc --edition 2021 .../main.rs -o /tmp/lesson_dev_env && /tmp/lesson_dev_env` 因 Python 3.10+ 检查失败退出 1。
- `npx -y tsx .../verify.ts` 因 Python 3.10+ 检查失败退出 1。
- 课程合同要求 code demo 自终止并在 canonical command 上 exit 0；配置诊断可以显示失败，但不应让课程本身失败。

影响：
- 学生在“第一课”运行诊断时容易把 expected diagnostic failure 理解成课程代码坏了。
- 本地网页或自动运行器如果执行该代码，会把环境未配置完成误判为课程运行错误。

### P2 - Dev Environment quiz 是旧分布

文件：
- `phases/00-setup-and-tooling/01-dev-environment/quiz.json`
- `phases/00-setup-and-tooling/01-dev-environment/quiz.zh-CN.json`

证据：
- 中英 quiz 都只有 5 道题。
- 阶段分布为 `pre=2/post=3`，没有 `check` 阶段。
- 课程合同要求 6 道题：1 pre、3 check、2 post。

影响：
- 站点可以渲染，但测验节奏不符合当前课程契约，也缺少课中检查。

### P2 - Dev Environment 代码缺少 tests 且入口命名不一致

文件：
- `phases/00-setup-and-tooling/01-dev-environment/code/`

证据：
- `code/` 有 `verify.py`、`verify.ts` 和 `main.rs`，但没有 `code/tests/`。
- `docs/en.md` 与 `docs/zh-CN.md` 声明 Python、Node.js/Rust，但只有 Rust 文件使用 `main.*` 命名；Python 和 TypeScript 入口不是 `main.py` / `main.ts`。

影响：
- 不符合每课至少 5 个 unit tests 的课程合同。
- 语言字段与 `main.*` 文件不一致，会让运行清单和网页运行面板更难稳定识别入口。

### P2 - 中文 Mermaid 标签仍是英文

文件：
- `phases/00-setup-and-tooling/01-dev-environment/docs/zh-CN.md`

证据：
- Mermaid 图示保留 `AI/ML Libraries`、`Language Runtimes`、`Package Managers`、`System Foundation`。

影响：
- 这是第一课的核心图。保留英文标签会削弱中文页面自然度。

### P2 - 本地网页运行面板仍指向旧入口

文件：
- `site/runtime-manifest.json`
- `site/lesson-assets.json`
- `scripts/detect_runtimes.py`
- `scripts/build_lesson_assets.py`

证据：
- 新增 `main.py` / `main.ts` 后，本地网页运行面板仍显示入口 `verify.py`。
- `runtime-manifest.json` 的第 00/01 课记录仍为 `entry: .../verify.py`。
- 重新生成 runtime 时发现本地模块 `verify` 会被当成外部 package；asset manifest 也会把 `code/tests/` 暴露到源码面板。

影响：
- 学生在网页上复制的运行命令不是新的稳定入口。
- 本地模块被显示成 package 会误导依赖判断，测试文件挤进源码面板也会降低可读性。

## 第 16 轮建议修复范围

1. 将 Python/TypeScript/Rust 验证脚本默认改为诊断后 exit 0，新增 `--strict` 模式在必需检查失败时退出 1。
2. 去掉 Python 验证脚本对不在 allowlist 中的 Matplotlib/Jupyter 必需导入；保留 NumPy 与 PyTorch/GPU 的诊断，其中 GPU 保持可选。
3. 补齐 `main.py` / `main.ts` 包装入口，并将 docs 的语言字段调整为 Python、TypeScript、Rust。
4. 将中英 quiz 收敛到 6 题、`pre=1/check=3/post=2`。
5. 补齐 stdlib unittest，覆盖 `run_check`、默认/strict 退出语义和 main wrapper。
6. 修复 runtime/asset 清单生成逻辑：本地 Python 模块不算外部 package，asset 面板排除 `code/tests/`，并刷新第 00/01 课清单。
7. 润色中文 Mermaid 标签，运行单课 demo、unittest、全量校验和本地网页复验。
