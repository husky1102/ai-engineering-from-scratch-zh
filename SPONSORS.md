`ai-engineering-from-scratch` 是一套免费的 MIT 许可课程。它包含 428 节课、20 个阶段，由 [Rohit Ghumare](https://github.com/rohitg00) 构建和维护。

赞助用于支付持续发布课程、保持网站运行、回复 issue 队列所需的时间。只接受现金。实物额度、股权或“我们来帮你写内容”这类安排不接受，原因见下方 [硬性规则](#硬性规则)。

如果你或你的公司想支持这套课程，这一页就是价目表。

## 硬性规则

这些规则不可协商。无法接受的赞助商会被礼貌拒绝。

1. **课程正文不放置广告。** Logo 只出现在 README sponsor block、课程网站 sponsor page 和 BACKERS.md。永远不会放在 `phases/**/docs/en.md`、`outputs/`、代码示例，或学习者正在阅读课程内容的任何位置。
2. **“Supported provider” 不等于 “recommended”。** 每节 API 课程都会在同一接口后展示三个或更多 provider。赞助商可以与其他 provider 并列出现，但绝不会被标成默认、首选，或“我该用哪个”的答案。
3. **不接受 sponsor-authored content。** 每节课都由维护者撰写。赞助商只能从技术准确性角度 review integration PR；他们不能提出叙事、框定 trade-off 或否决比较。
4. **不提供 roadmap veto。** Platinum 赞助商可以像任何人一样提交 roadmap 建议。最终发布什么由维护者决定。
5. **30 天 editorial-conflict exit。** 如果赞助商施压维护者偏置内容，赞助将在 30 天内终止并按比例退款。Logo 会在下一次 site deploy 中移除。
6. **冲突拒绝。** 如果赞助商产品直接违背课程原则（闭环 vibe-coding 工具、鼓吹 vendor lock-in、忽略 observability 或拒绝开放格式的 agent 产品），课程会拒绝赞助。拒绝由维护者自行判断。
7. **只接受现金。** 不接受实物额度、股权、免费硬件、“我们帮你做 DevRel”或捆绑交易。它们太容易被低估，也太难清晰入账。

## 潜在赞助商的反提案

如果你的公司有不同诉求，正确做法是先阅读档位和硬性规则，然后在第一封邮件里提出具体档位和期限。不要以“我们用免费额度换 hero placement 怎么样”或“我们想自己写 integration”开头——这些已经被上面的硬性规则预先拒绝，邮件最终只会收到本页链接。
